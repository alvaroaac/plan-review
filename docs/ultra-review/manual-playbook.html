ultra-review — human-as-orchestrator

main · 6545571 6 sessions paste & go

§ Architecture in one line

Separate Claude Code sessions, file-based handoff. Not Task subagents in one orchestrator window. Each session reads only what it needs, writes one JSON file, and exits. Reviewer and QA stages run in real parallel terminals.

ConcernOrchestrator + subagentsMulti-session (this playbook)
Context bloat4 reviewers + 4 QAs + merge ≈ 50–150 KB lives in one window foreverEach session ≈ diff + scoped files only
ParallelismSpawns subagents but observed serially4 terminals = real wall-clock parallel
Failure isolationOne bad subagent poisons orchestratorRe-run one session, others untouched
Maps to eventual codeNo — pipeline will be code-orchestratedYes — same on-disk JSON contract

📁 Scratch directory

One pinned timestamp per run. Everything writes here:

.ultra-review/manual/2026-05-15T1030/ ├── triage.json # step 1 ├── diff.patch # step 1 — `git diff <base>...HEAD` cached for reviewers ├── reviewer-R1.json # step 2 (× N reviewers, parallel) ├── reviewer-R2.json ├── reviewer-R3.json ├── reviewer-R4.json ├── qa-R1.json # step 3 (× N, parallel) ├── qa-R2.json ├── qa-R3.json ├── qa-R4.json ├── result.json # step 4 — matches §18c.1 RunResult schema └── qa-final.json # step 5 — optional polish pass

Export the timestamp once and substitute everywhere:

export TS=$(date -u +%Y-%m-%dT%H%M) mkdir -p .ultra-review/manual/$TS

Step 1 Triage — 1 session

Fresh Claude Code session in the repo. Reads the diff, picks 4 reviewer triples, resolves scoped files per the deterministic slicer rules.

You are the triage stage of ultra-review. Read docs/ultra-review/design-spec.md §4, §5, §13. The PR I want reviewed: <PR URL or describe branch>. Do: 1. `git diff <base>...HEAD > .ultra-review/manual/<ts>/diff.patch` and `git diff --stat`. 2. Inspect the diff. Pick exactly 4 reviewer triples from the role × depth × slice matrix in §4 that maximise differentiation. Drop slices that don't fit (e.g. docs-only PR drops security). 3. For each triple, resolve `scopedFiles` deterministically: - feature-area → files in the diff sharing the largest common dir - boundary-changes → files where exported symbols changed - risk-hotspots → files with the most insertions+deletions 4. Write `.ultra-review/manual/<ts>/triage.json` matching this shape: { "ts": "<ISO timestamp>", "repo": { ...RepoMetadata from §8... }, "triples": [ { "id": "R1", "role": "...", "depth": "...", "slice": "...", "scopedFiles": [...] }, ... ], "skipped": [ { "triple": {...}, "reason": "..." } ] } Print the chosen triples + reasoning, then stop. Do not run the reviewers.

Step 2 Reviewers — N parallel sessions

Open one fresh terminal per triple. Same prompt, change the reviewer id (R1 → R2 / R3 / R4). They never read each other's output.

You are reviewer R1 of ultra-review. Read docs/ultra-review/design-spec.md §4, §12.2 (reviewer prompt template), §8 (Finding type). Inputs: - `.ultra-review/manual/<ts>/triage.json` — find your triple by id "R1". - `.ultra-review/manual/<ts>/diff.patch` — the unified diff. - The files listed in your `scopedFiles`. If your depth is "deep", also read immediate dependencies and dependents (one-hop static `import`/`require` graph), capped at sane token budget. Do: 1. Review according to your role/depth/slice. Bias your finding categories per your role: - security: auth, injection, secrets, deserialization, supply chain - correctness: off-by-one, error handling, race conditions, type punning - architecture: layering, coupling, naming of boundaries, API contracts - dx-readability: naming, structure, dead code, comment quality 2. Write `.ultra-review/manual/<ts>/reviewer-R1.json` matching ReviewerOutput from §8: { "reviewerId": "R1", "spec": { ...your full ReviewerSpec... }, "findings": [ Finding, Finding, ... ], "notes": "freeform rationale for the senior" } Each Finding must have: id (F001…), reviewerId, severity, title (≤80c), body (markdown), file, startLine, endLine, suggestion (markdown fenced or null), tags array. Do not invent findings to pad the review. Zero findings is a valid output.
Tip: open all 4 terminals first, paste each, then walk away. Doing one at a time defeats the parallelism.

Step 3 QA roast — N parallel sessions

Adversarial pass. One session per reviewer output. Goal: break the review, not the code. Run after step 2 finishes; parallel across reviewers.

You are the QA gate for reviewer R1's output. Be adversarial. Your job is to roast this review — find what's wrong with it, not what's wrong with the code. Inputs: - `.ultra-review/manual/<ts>/reviewer-R1.json` - `.ultra-review/manual/<ts>/diff.patch` - The files cited in R1's findings (read them directly to verify claims). Check each finding for: 1. **Factually wrong** — does the code actually do what the finding claims? 2. **Line numbers wrong** — do startLine/endLine point at the right span? 3. **Severity inflation** — is a "critical" actually a nit? 4. **Hallucinated APIs** — does the suggestion reference functions/types that don't exist? 5. **Scope creep** — does the finding criticise code outside the diff? 6. **Duplicate of the diff's own intent** — does the finding complain about something the PR is explicitly fixing? 7. **Missed obvious issues** — given R1's role/depth/slice, what should they have caught and didn't? Write `.ultra-review/manual/<ts>/qa-R1.json`: { "reviewerId": "R1", "verdicts": [ { "findingId": "F001", "verdict": "valid" | "refute" | "downgrade" | "rephrase", "reason": "...", "newSeverity": "..." (if downgrade) } ], "missed": [ { "what": "...", "where": "file:line", "severity": "..." } ] }

Step 4 Senior merge — 1 session

Reads everything, dedupes, applies QA verdicts, writes the rich exec summary, picks the verdict tag, emits the final result.json in §18c.1 shape.

You are the senior synthesis stage of ultra-review. Read docs/ultra-review/design-spec.md §12.3 (senior prompt template), §8 (SeniorSynthesis, RunResult), §18c.1 (RunResult JSON template). Inputs: - `.ultra-review/manual/<ts>/triage.json` - `.ultra-review/manual/<ts>/reviewer-R*.json` (all of them) - `.ultra-review/manual/<ts>/qa-R*.json` (all of them) - `.ultra-review/manual/<ts>/diff.patch` Do: 1. Apply QA verdicts: drop refuted findings (set `refutedBy`), apply severity downgrades, rephrase where requested. 2. Dedupe: when ≥2 reviewers raised the same finding, keep one, set `agreedWith` to the other reviewer IDs. 3. Merge in any "missed" findings from QA only if they're concrete (file:line cited) and the QA gate is confident. 4. Sort findings by severity (critical → major → minor → nit), stable within severity. 5. Write an executive summary in markdown. Rich, not perfunctory: - 1–2 sentence headline of what the PR is doing - The top 2–3 findings called out with their gist - Net verdict in plain English - Counts by severity 6. Pick a verdict tag: `block` | `request-changes` | `approve-with-nits` | `approve`. 7. Write `.ultra-review/manual/<ts>/result.json` matching the populated example in §18c.1 exactly. Include: - `repo` and `meta` (from triage.json + your timing — set finishedAt = now) - `synthesis` with findings, summary, verdict - `reviewers` with each reviewer's full spec + their original (pre-dedupe) findings preserved for debugging - `synthesis.htmlFragments` is optional — skip it for the manual demo, the formatter will fall back to defaults Validate the JSON parses cleanly before exiting.

Step 5 Final QA — 1 session, optional

One last adversarial pass on the synthesised report itself.

You are the final QA gate. Read .ultra-review/manual/<ts>/result.json and roast the synthesis itself. Check: - Does the executive summary actually match the findings? (No "approves" when there's a critical, no "blocks" when there are only nits.) - Are deduped findings genuinely the same? (No false-merge of distinct issues.) - Is the verdict tag calibrated against the severity counts? - Any finding whose `refutedBy` reason is weak — should it come back? - Is the summary rich enough to stand alone, or does it read like filler? Write `.ultra-review/manual/<ts>/qa-final.json`: { "verdict": "ship" | "revise", "issues": [ { "what": "...", "fix": "..." } ], "summaryFeedback": "..." } If verdict is "revise", re-open the merge session, apply the fixes, write a new result.json.

Step 6 Format

Once Plan 2 ships:

ultra-review format .ultra-review/manual/<ts>/result.json --out .ultra-review/manual/<ts>/report.html

Until then, paste result.json into a Claude session with the §11 V4 Glass HTML template inlined and have it render the report by hand.

⚙ Practical notes

  • Open all 4 reviewer terminals before pasting anything. Pasting serially defeats the parallelism.
  • Use cd worktrees, not separate clones — same .git dir, separate working trees. git worktree add ../ultra-review-r1 HEAD. Or just cd four terminals into the same repo — only the reviewer sessions read files, they don't write outside the scratch dir.
  • Pin one timestamp. export TS=$(date -u +%Y-%m-%dT%H%M) and substitute in every prompt.
  • Skip step 3 (QA roast) on the first run to baseline what the reviewers tend to hallucinate. Add it back once you've calibrated.
  • Skip step 5 (final QA) for short PRs. Under ~200 LOC the diminishing returns aren't worth the round trip.
